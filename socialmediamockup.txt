I spent the last two years using AI coding agents almost every day.

The biggest lesson:

Prompt engineering and individual agent sessions are not the durable layer anymore.

The new engineering frontier is:

- agent skills
- repo-local instructions
- orchestration
- agent harness optimization
- long-running single-prompt buildouts
- forcing agents to finish the whole task and review their own work before handoff

So I put the system I use into a public GitHub repo:

Agent Skill Architecture Guide

It includes:

- `AGENTS.md`: the operating rules I use for agent behavior, Git hygiene, contextual editing, implementation review, and handoff discipline.
- `agent-copy`: a skill that copies your canonical global instructions into any repo and adds repo-specific guidance.
- `new-skill-builder`: a skill for creating, installing, validating, and auditing reusable agent skills.
- `new-mcp-builder`: a skill for planning and building MCP server integrations.

The top half of `AGENTS.md` handles the stuff that kept breaking my workflows:

- no generated branch mess
- no context mangling
- no yes-man loops
- no reflex apology theater
- no pausing after every section of a long prompt
- mandatory outside-review before final handoff
- clearer boundaries around Git, Markdown editing, testing, and repo-specific instructions

The `agent-copy` workflow is the part that makes this practical.

Instead of manually copying an `AGENTS.md` into every new project, install the skill once, open any repo, and ask:

Use agent-copy to create or refresh this repo's AGENTS.md from my canonical global instructions, then add repo-specific guidance based on the files in this project.

The default source is `~/.codex/AGENTS.md`, but the architecture is not Codex-only:

canonical global instructions -> repo-local AGENTS.md -> repo-specific tail

If your canonical instructions live somewhere else, point `agent-copy` at that file.

That gives the agent:

- your global behavior rules
- the repo's stack
- local commands
- branch workflow
- build/test flow
- project-specific boundaries
- implementation notes
